### 2026-05-08 — Don't extrapolate scope from the project name; do a small intake first
- **What happened**: Project directory was `cisco-ntp-dns`. On `/framework-init` and again on `/session-start`, I assumed a multi-device Cisco config-fix tool and produced a heavy plan with 18+ scoping questions covering inventory sources, multi-platform fanout, change windows, rollback strategies, and TACACS credentials. The actual scope was a single-device bench-test rig that simulates NTP and DNS for one Neeve gateway over a direct Cat6 cable. The first plan was discarded entirely; the user then wrote out the real scope in plain prose.
- **Correction**: For greenfield projects (no source code yet, user has signaled "still defining the problem"), the FIRST scoping action must be a 3–5 question intake that establishes (1) target system count and topology, (2) what's known vs. unknown, (3) what success looks like. Only after those answers is detailed plan content worth writing.
- **Rule**: For greenfield/scoping sessions, ask a 3–5 question intake before drafting more than a one-paragraph plan stub. Never extrapolate scope from the directory name or PROJECT_CONFIG defaults — confirm with the user.

### 2026-05-09 — Don't claim runtime behavior in code comments without testing it
- **What happened**: The chunk 4+5 commit (`0dcc4b6`) wrote `docker-compose.yml` with the comment "No cap_add — chrony.conf uses `manual` (no clock discipline), so CAP_SYS_TIME is not needed." That claim was untested — the rig was never actually run end-to-end before the commit. On first hardware bring-up, the ntp container crash-looped: chrony 4.5 calls `adjtimex(0x8001)` during privdrop init regardless of `manual` mode, and dies with "Operation not permitted" without `CAP_SYS_TIME`. Wasted about 10 minutes on diagnosis when a 2-minute test run before the original commit would have caught it.
- **Correction**: Don't write declarative comments about WHY something is omitted (caps, deps, error handlers) unless you've validated empirically. If you're tempted to write "X is not needed because Y," either (a) prove it with an experiment, or (b) leave the comment out and let the code stand on its own. Speculative comments rot into misinformation that future maintainers trust.
- **Rule**: Comments that justify omissions ("no cap_add needed because…", "no error handling needed because…") require empirical proof attached to the same commit that introduces them. If the rig wasn't deployed, don't claim what it doesn't need.

### 2026-05-09 — `ssh -t` doesn't allocate a TTY when invoked through Claude Code's `!` prefix
- **What happened**: For the passwordless-sudo setup on the VM, I told the user to run `! ssh -t gh0stwhee1@host "echo ... | sudo tee ..."`. The `!` prefix in Claude Code runs the command through a non-interactive harness with no TTY on stdin, so `ssh -t`'s pty allocation is rejected ("Pseudo-terminal will not be allocated because stdin is not a terminal"). sudo then can't prompt for a password and dies "A terminal is required to authenticate." We went through three failed attempts (variants on quoting and `-t` flags) before the user opened a real PowerShell window outside Claude Code and ran ssh interactively there.
- **Correction**: When a remote command needs an interactive prompt (sudo password, ssh-agent passphrase, etc.), `!` in Claude Code is the wrong harness — it has no TTY. Either: (a) use passwordless mechanisms set up in advance, (b) tell the user to open a real terminal outside Claude Code and paste the command there, OR (c) for SSH specifically, have the user `ssh` interactively (no remote command), then paste a multi-line block on the remote shell. Don't guide users through `! ssh -t ...` for password prompts.
- **Rule**: The `!` shell-out in Claude Code is non-interactive — assume no TTY. For commands that prompt (sudo, ssh-add, gpg, etc.), instruct the user to use a real terminal window or set up auth non-interactively first.

### 2026-05-09 — Verify "earlier session" claims before accepting them as fact
- **What happened**: User said "we will resume once we have the specific NTP server URL and the 2 DNS IP addresses we need to use. I had provided these in an earlier session, are you able to locate these?" I did the search and found the IPs were never recorded — Q-DNS has been listed as "TBD — supply / extract from node config" since the very first scoping commit, and no transcripts contain user-provided values. The NTP server (`ntp2.wbg.org`) WAS recorded; the DNS IPs were not. Without searching, I'd have ended up looking like I "lost" data the user expected to be there.
- **Correction**: When a user references information from a prior session ("I told you X earlier", "we already discussed Y"), confirm by searching session files, plans, transcripts, and memory before either accepting it as known OR letting the user assume it's known. The cost of the search is small; the cost of acting on a phantom fact (or accidentally implying you have one when you don't) is much larger.
- **Rule**: For "earlier session" references, search and verify before treating the data as in-hand. Tell the user what you found and what you didn't — don't fill the gap with assumptions.

### 2026-05-09 — When DHCP is healthy and the device is still silent, the rig is done; pause and consult operator
- **What happened**: Across this session we landed three rig-side changes (`fix(setup-host-nic)`, `feat(dhcp): DHCP_DOMAIN_NAME`, plus `.env` updates with operator-disclosed DNS IPs) and ran multiple captures. The Advantech 2484's DHCP behavior was textbook clean throughout — DISCOVER → OFFER → REQUEST → ACK in milliseconds, unicast T1 renewal at 5 min, ARP-responsive in both directions. But across ~10 min of cumulative observation the device emitted **zero DNS, zero NTP, zero spontaneous packets**. The temptation was to keep iterating (try option 120 with a placeholder, run longer captures, tune another DHCP option). Instead we recognized the rig had completed its job and the wedge was somewhere we couldn't see from the wire.
- **Correction**: When a DHCP client's full lease lifecycle (DISCOVER → ACK → T1 renewal) is observably healthy AND the device responds to ARP, the rig is functionally correct. Further blind probing (sending DHCP options the device hasn't proven it requires, or stretching capture windows past the point of diminishing returns) wastes time without adding information. The cheap next steps at that point are operator-side: get docs on the device's expected boot-init behavior, attach a serial/console tap during boot, or get a specific value to test from someone with firmware visibility. Don't continue rig-side experiments hoping for a different result.
- **Rule**: If DHCP-state-machine + ARP are both healthy but zero application packets appear in 5+ min of observation, classify the blocker as device-side and pause rig changes until you have operator/firmware info. Record the rig as "functionally complete and correct" so future agents don't restart the iterate-the-rig loop.

### 2026-05-12 — Prove rig correctness with a known-good substitute client, don't infer it by elimination
- **What happened**: After 2026-05-09 classified the chunk-4 silence as "device-side" based on the rig's symmetric correctness (DHCP healthy, ARP responsive, device silent), we still lacked an *affirmative* proof that the rig itself answers correctly to a real DHCP/DNS/NTP exchange. In this session we plugged a Windows laptop into the same cable as the Advantech 2484 with Wi-Fi off, the laptop DHCP'd into the exact lease shape `.env` specifies, both /32-aliased DNS IPs returned the seeded `ntp2.wbg.org → 192.168.50.123` answer, and `w32tm /stripchart` round-tripped with chrony. Then we swapped the device back in and the chunk-4 silence reproduced exactly (0 DNS, 0 NTP, only ARP replies). Net: rig is *provably* correct, device is *reproducibly* silent. Stronger evidence than the 2026-05-09 single-sided inference.
- **Correction**: When a device-under-test exhibits behavior the rig can't diagnose, don't just classify by elimination — affirmatively prove rig correctness with a substitution test using a known-good RFC-compliant client (Windows/Linux laptop, second NIC, virtual client) on the same physical link. Two-sided proof beats single-sided inference and gives future agents a stamp they can trust without re-running the experiment.
- **Rule**: Before resting on a "blocker is device-side" classification, run a substitution test with a known-good client on the same physical link and capture: (a) the wire-level pcap, (b) a one-page summary alongside the pcap, (c) a corresponding device-side reproduction in the same session. Land the artifacts in `out/run/<test>-<timestamp>.{pcap,summary.md}` so the evidence is self-describing on the rig.
